The Big Picture

What is this paper about — in one paragraph?

Optical networks are the fiber-optic highways that carry virtually all internet traffic between data centres, cities, and continents. Running them today requires engineers to write complicated, device-specific control scripts by hand — one for each type of equipment, in the manufacturer's own language. This paper asks: what if you could just type plain English and have an AI automatically control all the hardware? The answer is AgentOptics — a system that connects a large language model (LLM) to 64 standardized control "tools" covering 8 real optical devices, achieving 99% task success on 410 real hardware tasks.

Analogy: Imagine you walk into a recording studio with 20 different mixing desks, amplifiers, and signal processors — all from different brands with different manuals. Today, you'd need an expert for each machine. AgentOptics is like hiring one super-translator who reads all the manuals, listens to what you want in plain English, and operates all the machines for you — correctly, every time.
Before You Start

Prerequisites to understand this paper

You don't need a PhD in optics to follow this — but a few core concepts make everything click.

01 · Networking Basics
Optical Networks & Fiber
Light pulses carry data through glass fibers. Key devices: amplifiers (boost signal), ROADMs (route different wavelengths/colors), transceivers (convert electrical to optical). Each fiber can carry dozens of parallel "channels" at different wavelengths — this is called WDM (Wavelength Division Multiplexing).
Know: ROADM, EDFA, OSA, WDM
02 · Software-Defined Networking
SDN & Control Planes
SDN separates "what to do" (control plane) from "how to physically do it" (data plane). A central controller sends commands to devices via protocols like NETCONF/SSH or vendor APIs. This was the state-of-the-art before LLMs entered the picture.
Know: NETCONF, SDN controller, northbound API
03 · Large Language Models
How LLMs Work
LLMs (like GPT, Claude, Gemini) are AI systems trained on massive text corpora. They can understand and generate language, follow instructions, reason step-by-step, and call external functions (tools). The key insight: they can translate between human intent and structured actions.
Know: GPT, Claude, tool-calling, tokens
04 · Model Context Protocol
MCP — The Bridge
MCP is an open standard (released 2024) that defines how LLMs can discover and call external tools safely. Think of it as a USB standard for AI tools — any LLM can plug in to any MCP server and use its tools, without knowing the underlying implementation. This paper builds everything on MCP.
Know: MCP client, MCP server, tool schema
05 · Agentic AI
Agents vs. Chatbots
A chatbot answers questions. An agent takes actions in the world — calling tools, running loops, making decisions based on intermediate results, and adapting. AgentOptics is an agent: it doesn't just describe what to do, it actually does it on real hardware.
Know: tool-calling, planning, closed-loop
06 · Code Generation vs. Tool Calling
Two Ways to Use LLMs for Control
The old way: ask LLM to write Python code → execute the code → hope for no errors. The new way (this paper): give LLM a structured menu of validated tools → LLM picks and calls tools directly → no raw code, no import errors, no syntax bugs. This is the core difference.
Know: LoRA fine-tuning, code execution risks
Historical Context

How we got here — the evolution of optical network control

Every research paper builds on a lineage. Here's the full story from manual scripts to AgentOptics, with each era's limitations that motivated the next step.

Era 1 · 1990s–2010s
Manual CLI & Custom Scripts
Engineers SSHed into each device and ran vendor-specific CLI commands. To automate, teams wrote Python/Bash scripts reading device manuals page by page. Every device brand had a different interface: Lumentum ROADMs used NETCONF, CFP2 transceivers used custom APIs, OSAs used PyApex. Each required bespoke code.
⚠ Limitations: Huge onboarding effort per device. Zero portability across vendors. Any firmware update could break scripts. Couldn't scale to heterogeneous networks.
Era 2 · 2010s–2020
Software-Defined Networking (SDN)
SDN moved control logic to a centralized controller. Standards like OpenROADM defined common data models (YANG schemas). Tools like NETCONF/RESTCONF made device communication more uniform. This was a huge leap — operators could control multi-vendor gear from one place.
⚠ Limitations: Still required expert engineers to write SDN apps and northbound APIs. Multi-vendor interoperability was inconsistent. Adding a new device type still required significant engineering. Physical-layer details were poorly abstracted.
Era 3 · 2022–2024
LLM-Assisted Control (CodeGen approach)
LLMs like GPT-4 could read device manuals or reference code and generate Python scripts to control hardware. AlarmGPT automated alarm handling. Other systems embedded API specs in prompts and asked GPT to generate SDN calls. Fine-tuned local models (like CodeLlama) were trained on device-specific code datasets.
⚠ Limitations: Generated code often imported nonexistent libraries, called wrong function names, had syntax errors. Required huge prompt context (entire manuals). Fine-tuned models overfit — they failed when users paraphrased requests. No standardized interface across devices. Success rate: only 9–50%.
Era 4 · 2025–2026 (This Paper)
MCP-Based Agentic Control — AgentOptics
Instead of generating code, the LLM selects and calls pre-built, validated tools exposed through the Model Context Protocol. The LLM reasons; standardized MCP tools execute. No raw code generation. No device manuals in the prompt. No fine-tuning needed per device. Works across 8 devices, multiple LLM backends, 99% success rate.
System Design

How AgentOptics actually works — step by step

The architecture has three clean layers: the user's intent → LLM reasoning → MCP tool execution on real hardware. Here's each step in detail.

01
User types a plain-English command
No scripting knowledge required. The user simply describes what they want — just like asking a knowledgeable colleague. The system accepts many phrasings of the same intent (tested with paraphrasing variants).
"What's the current gain of the EDFA amplifier on ROADM1?"
02
MCP Client receives the request and queries the LLM
The host application (running the MCP client) forwards the task to the LLM — either a cloud model like GPT-5 or Claude Sonnet, or a local open-source model like Qwen-14B running on a GPU server. The LLM receives the user's text plus the catalog of available MCP servers and their descriptions.
03
LLM identifies the right MCP server(s) and tools
The LLM reasons: "This is about an EDFA on a ROADM, so I need the Lumentum ROADM MCP server." It then reads that server's tool list and picks the specific tool. It figures out any required parameters (like which EDFA port, which gain value). It does NOT write code — it fills in the tool schema.
Selected tool: get_edfa_info(device="roadm1", port="booster")
04
MCP Server executes the tool on the real device
The MCP server for that device receives the tool call, validates parameters (rejecting bad values like out-of-range wavelengths), calls the actual device API (NETCONF, vendor Python SDK, etc.), and collects the real hardware response. No LLM involvement here — pure deterministic execution.
Device returns: {"gain_dB": 18.5, "status": "OK", "input_power": -12.3}
05
LLM formats a human-readable response
The raw JSON is returned to the LLM, which converts it into a natural language answer for the user. For multi-step tasks, the LLM loops back — each tool result feeds the next decision. This is the closed-loop agentic behavior that enables complex orchestration.
"The EDFA booster on ROADM1 is operating at 18.5 dB gain with input power −12.3 dBm. Status is normal."

Real-World Demos

5 Case Studies on Real Hardware

1
DWDM + 5G Monitoring
Link Setup & Performance Monitoring
AgentOptics sets up a full DWDM link — routing a 400 GbE coherent signal and an analog radio-over-fiber (ARoF) signal across two ROADMs and 20 km of fiber. It then monitors both channels simultaneously. 9 tools invoked, 4 devices coordinated in one natural language session.
→ OSNR: 32.6 dB · ARoF EVM: 3.91%
2
5G ARoF Optimization
Autonomous Bias Voltage Sweep
For a 5G NR fronthaul link, AgentOptics sweeps the ARoF transmitter bias voltage from −1.5V to 0V in 0.1V steps, measures SNR and BER for QPSK, 16QAM, and 64QAM modulations, and identifies the optimal operating point — all from a single English prompt.
→ Optimal bias found autonomously across 3 modulations
3
BER Optimization
Multi-Span Channel Power Tuning
In a 2-span link (20 km + 27 km field fiber), AgentOptics adds a new 400 GbE channel at 195.5 THz and iteratively adjusts transmit power in 2 dB steps until pre-FEC BER is minimized — without degrading existing channels (±0.5 dB constraint enforced autonomously).
→ BER converged to 5×10⁻⁴ over 5 iterations
4
Quantum-relevant Control
Closed-Loop Polarization Stabilization
Polarization drift (caused by temperature, vibration) degrades coherent and quantum links. AgentOptics stabilizes polarization to a target state using gradient descent over 4 piezo actuators — and recovers from deliberate fiber disturbances mid-optimization without any human intervention.
→ Converged in 12 iterations, 0.23 sec/iteration
5
AI-Assisted Sensing
DAS Fiber Cut Detection
Distributed Acoustic Sensing (DAS) monitors fiber vibration patterns. AgentOptics controls the DAS interrogator, retrieves waterfall plot images, and uses LLM vision + prompt engineering to classify whether a fiber cut is imminent — distinguishing noise, finger agitation, and real fiber cuts.
→ Correct classification on real 53 km field fiber cut event
Critical Analysis

Limitations — what the paper doesn't solve

No paper is perfect. Understanding these gaps is key to identifying where new research can contribute.

⚠
Weak error detection (76.7% worst case)
When a user submits a command with a missing or invalid value (e.g. "set wavelength to 0 nm"), the system should refuse. But Claude Haiku 3.5 only caught these correctly 76.7% of the time. The LLM sometimes still invokes a tool with bad parameters instead of rejecting the request. This is a safety concern for real networks.
⚠
Small local models break on complex tasks
Qwen-0.6B drops to 70% on 3-action tasks. Small models struggle to reliably select the right tool from a large catalog when multiple tools are needed in sequence. Multi-step coordination is fundamentally harder with low-parameter models. This limits privacy-preserving local deployments.
◈
Tool naming confusion between similar tools
AgentOptics sometimes calls osa_set_center_wavelength when it should call osa_set_start_wavelength. When tool names are semantically similar, the LLM can pick the wrong one. Fixing this requires better tool naming design or disambiguation hints in tool descriptions.
◈
Higher latency than CodeGen (11–24 sec per task)
Because AgentOptics involves multiple back-and-forth rounds between the MCP client and LLM (tool discovery → tool selection → execution → result processing), each task takes longer than simple code generation (8.6 sec). For real-time network control scenarios, this round-trip overhead may be unacceptable.
◈
Large spectral data bypasses the agent
OSA spectrum measurements return large numerical arrays (thousands of data points). These were deliberately excluded from the AgentOptics workflow because they would overflow the LLM context window and dramatically increase token costs. The agent configured the OSA, but spectrum acquisition was done separately. This is a significant architectural gap.
○
Only 8 devices tested — small lab, not real carrier network
The testbed has 8 device types. A real optical carrier network might have hundreds of device models from dozens of vendors, thousands of network elements, and millions of possible tool combinations. Whether the MCP architecture scales — and how tool retrieval accuracy degrades with a huge tool catalog — is completely untested.
○
No security or adversarial analysis
What happens if a malicious user prompt tries to disable amplifiers, overdrive laser power, or drop all fiber connections? The paper doesn't evaluate safety against adversarial inputs. In production networks, unauthorized device control is a critical security risk that needs dedicated guardrail mechanisms — none are proposed here.
○
Reactive rather than proactive
AgentOptics responds to requests and optimizes on demand. It doesn't monitor network telemetry continuously and proactively predict or prevent degradation. The DAS case study detects events after they happen. Truly autonomous network management requires predictive, always-on closed-loop intelligence — not present here.
Future Research Landscape

Scope of this field — where the next papers should go

This paper opens more doors than it closes. Here are the richest research directions, each with a clear gap, why it matters, and what a new paper would need to prove.

DIRECTION 01 · Safety
Guardrail-Protected Agentic Control
Design a safety layer that sits between the LLM and MCP tools. It should validate parameter ranges, enforce access-control policies (user roles), rate-limit destructive operations, and reject adversarial prompts. Evaluate against red-team attacks on real optical hardware. This is completely absent in existing literature.
Novelty: First safety-aware MCP framework for optical control
DIRECTION 02 · Scalability
100+ Devices, 1000+ Tools
Test MCP with massive tool catalogs using hierarchical server organization (meta-server routes to domain servers). Measure how tool retrieval accuracy degrades. Introduce semantic search / embedding-based tool selection instead of feeding all tools to the LLM. Benchmark on a simulated carrier-grade network.
Novelty: First systematic scalability study for MCP-based control
DIRECTION 03 · Intelligence
Proactive Fault Prediction
Extend from reactive control to always-on monitoring. Feed streaming telemetry (OSNR, BER, power levels) into a continuous LLM reasoning loop. Predict fiber faults, amplifier degradation, and polarization drift before they impact traffic. Combine with DAS, optical power monitors, and coherent DSP feedback.
Novelty: Streaming LLM-in-the-loop proactive network management
DIRECTION 04 · Multi-Agent
Specialized Agent Teams
Split the single LLM into a team: an Amplifier Agent, a Switching Agent, a Sensing Agent, and an Orchestrator. Each specializes in its domain. They communicate using structured protocols when a task crosses boundaries. Compare coordination overhead vs. single-agent accuracy on long-horizon 10+ device tasks.
Novelty: Multi-agent hierarchical control for optical infrastructure
DIRECTION 05 · Efficiency
Latency-Optimized Agentic Control
Current round-trip latency (11–24 sec) is too slow for real-time network events. Research: parallel tool execution, tool result caching, speculative tool pre-selection, lightweight local models for routine tasks with cloud escalation only for complex ones. Target: sub-second response for single-action tasks.
Novelty: Real-time agentic control for dynamic optical networks
DIRECTION 06 · Efficiency
Large Spectral Data Integration
The paper excluded full OSA spectrum arrays due to token cost. Design a compression pipeline: local spectrum summarization (peak detection, anomaly flags, OSNR estimates) → compact structured summary → LLM reasoning over summary. Enable full closed-loop spectral optimization within the agent workflow.
Novelty: First in-context spectral data reasoning for agentic control
DIRECTION 07 · Quantum
Quantum + Classical Hybrid Control
The paper briefly mentions quantum photonics. Extend AgentOptics to control quantum hardware: entangled photon sources, beam-splitters, single-photon detectors, polarization analyzers for QKD. Test whether MCP-based abstraction generalizes from classical to quantum optical devices. Critical for quantum network deployments.
Novelty: First agentic control framework for quantum optical networks
DIRECTION 08 · Models
Domain-Optimized Small Models
Qwen-0.6B hits 70% — can we do better with smarter training? Instead of LoRA on raw code (which overfits), train on MCP tool-call trajectories: full (user intent → tool sequence → result) traces. Test generalization to unseen device types and paraphrased inputs. Enable fully private, edge-deployable network AI.
Novelty: Tool-trajectory fine-tuning for generalized optical control
Research Guidance

How to create a paper in this field

Whether you want to extend AgentOptics or build something adjacent, here's a concrete roadmap for producing a publishable research paper in agentic AI for optical/networked systems.

1
Pick a specific, unsolved problem
Don't try to do everything. From the scope section above, pick ONE direction. Define the exact gap: "AgentOptics achieves X but fails at Y because of Z." Your paper's contribution is a system/method that addresses Z. The clearer the gap statement, the stronger the paper.
Example gap: "AgentOptics cannot handle spectral data in-context due to token cost limits. We propose SpecSumMCP, a compression pipeline that reduces spectrum arrays to structured summaries, enabling inline reasoning."
2
Survey the related work thoroughly
Read all 56 references in AgentOptics, plus recent papers citing it. Know: MCP specification, agentic AI frameworks (ReAct, HuggingGPT, SWE-agent), optical SDN literature (OpenROADM, GNPy), LLM fine-tuning methods (LoRA), and existing optical AI papers (AlarmGPT, AutoLight). Build a table of what each does and what it misses.
3
Design your system architecture
Draw the end-to-end flow before writing any code. Define: which LLMs you'll use, what new MCP tools you'll build, what devices you'll control, how your method differs from AgentOptics. Identify your key design choices and explain why each one matters — this becomes Section III of your paper.
AgentOptics built 8 MCP servers, each with a clear device scope. Follow this pattern — one server per device family, with clean tool naming conventions.
4
Build a rigorous benchmark
AgentOptics built 410 tasks across 5 variant types on real hardware — this is the gold standard. For your paper: define task categories, difficulty levels, and robustness variants. Use real devices whenever possible (or high-fidelity simulators). Establish ground truth using manually validated scripts. This section is what reviewers will scrutinize most.
Aim for 100+ tasks minimum. Include at least: basic functionality, paraphrasing robustness, error handling, and multi-step coordination. Run everything on real hardware if you can.
5
Choose meaningful baselines
Always compare against: (1) the prior state of the art (CodeGen, manual scripts), (2) AgentOptics vanilla if you're extending it, and (3) ablated versions of your own system (e.g. without your key contribution). This lets reviewers see exactly how much your contribution adds. Use the same evaluation metric (success rate) for fair comparison.
6
Run case studies on real systems
AgentOptics demonstrates 5 real-world scenarios. Case studies make a paper compelling because they show the system works end-to-end in realistic conditions. Pick 2–3 scenarios that showcase your unique contribution — not just that your system works, but that it does something the prior work couldn't. Measure: success rate, cost, latency, and convergence quality.
Access to real optical testbeds is a key differentiator. Partner with network labs, telecom companies (NEC Labs, Nokia Bell Labs, Ciena), or university COSMOS/FIBER testbeds.
7
Target the right venue
Top venues for this field: IEEE/Optica OFC (Optical Fiber Communications Conference), IEEE INFOCOM, ECOC (European Conference on Optical Communications), Journal of Optical Communications and Networking (JOCN), IEEE Journal of Lightwave Technology. For AI-heavy contributions: NeurIPS, ICLR, or MLSys workshops on AI for networking are also appropriate.
AgentOptics was submitted to IEEE — likely JOCN or Journal of Lightwave Technology. These are the flagship venues for this intersection.
8
Open-source your code and benchmark
AgentOptics released all code and benchmark tasks on GitHub. Open-sourcing your work dramatically increases impact, citations, and reproducibility credibility with reviewers. Include: all MCP server implementations, benchmark task files, evaluation scripts, and instructions to reproduce your results. This is increasingly expected at top venues.
Reference

Key terms glossary

ROADM
Reconfigurable Optical Add-Drop Multiplexer. Routes specific wavelengths of light through a fiber network, like a smart switchboard for light.
EDFA
Erbium-Doped Fiber Amplifier. Boosts optical signal strength using a doped fiber excited by a pump laser. No electrical conversion needed.
OSA
Optical Spectrum Analyzer. Measures the power of light at each wavelength — like a "spectrum camera" for fiber signals.
CFP2-DCO
A pluggable coherent 400 GbE transceiver. Sends/receives high-speed data using advanced modulation (PM-16QAM) over a single wavelength.
WSS
Wavelength Selective Switch. Inside a ROADM, routes individual wavelength channels to different output ports with programmable attenuation.
MCP
Model Context Protocol. An open standard that lets LLMs discover and call external tools through a client-server architecture with defined schemas.
ARoF
Analog Radio-over-Fiber. Transmits wireless RF signals (like 5G) over optical fiber, converting between electrical and optical domains.
DAS
Distributed Acoustic Sensing. Uses backscattered light in fiber to detect vibrations along the cable — can sense footsteps, vehicles, and fiber cuts.
DWDM
Dense Wavelength Division Multiplexing. Carries many data channels simultaneously on one fiber by using different wavelengths (colors) of light for each.
Pre-FEC BER
Bit Error Rate before Forward Error Correction. A measure of signal quality — lower is better. Target: below 1×10⁻³ for 400 GbE.
LoRA
Low-Rank Adaptation. A parameter-efficient fine-tuning technique that trains a small set of extra weights instead of the whole model.
OSNR
Optical Signal-to-Noise Ratio. Ratio of signal power to noise power in dB. Higher is better; 32+ dB is excellent for 400 GbE links.
